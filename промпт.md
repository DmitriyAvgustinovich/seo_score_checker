## Manifest / permissions guardrails

Keep Manifest V3.

In manifest.json:

- keep `"manifest_version": 3`
- keep minimal permissions
- prefer only:
  - `"activeTab"`
  - `"scripting"`

Do not add:

- `"<all_urls>"`
- broad `host_permissions`
- `"tabs"`
- `"identity"`
- `"cookies"`
- `"webRequest"`
- `"webRequestBlocking"`
- `"debugger"`
- `"history"`
- `"bookmarks"`
- `"storage"` unless there is a clear, minimal reason
- remote code
- externally hosted JS/WASM
- declarative content scripts with `matches: ["<all_urls>"]`

If the current project already has broad permissions or `<all_urls>`:

1. Check whether they are actually required.
2. If not required, remove them.
3. Refactor toward user-triggered `chrome.scripting.executeScript` on the active tab.
4. Explain the before/after permissions in the final changelog.

Use `activeTab + scripting` for current-page analysis after the user clicks the extension.

Do not add a background crawler or sitewide scanner.

## Top 3 fixes: explainable format

Refactor Top 3 fixes so each fix includes:

- Issue
- Evidence
- Why it matters
- Fix
- Impact / score loss
- Confidence

Example:

Issue:
Meta description is missing

Evidence:
No meta description tag was found on this page.

Why it matters:
Search engines may create a less controlled snippet.

Fix:
Add a clear 120–160 character meta description.

Impact:
-8 points

Confidence:
High

Rules:

1. Top fixes must be generated from actual DOM evidence.
2. Do not show vague generic advice as a top fix.
3. Do not let low-confidence checks outrank high-confidence SEO issues.
4. Do not show social-only issues as High risk.
5. If there are fewer than 3 meaningful fixes, show fewer than 3 and add a calm message like:
   No more high-priority fixes found in this quick current-page check.

## Scoring changes

The score must not be a simple warning count.

Use weighted, explainable scoring.

If there is already a scoring model, refactor it instead of replacing everything blindly.

High-impact checks should drive the score:

- noindex / indexability
- canonical
- title
- meta description
- H1
- heading structure
- viewport
- image alt summary
- schema presence / JSON-LD validity

Lower-impact checks should be secondary:

- Open Graph
- Twitter tags
- lang
- charset
- raw link counts
- social metadata

Suggested scoring structure:

Indexability: 25 points
- noindex is a critical issue
- invalid robots directive is high priority
- canonical problems are high priority

Metadata: 20 points
- title presence
- title clarity/length
- meta description presence
- meta description clarity/length

Headings: 15 points
- H1 presence
- multiple H1s
- basic heading order

Technical basics: 10 points
- viewport
- charset
- lang
- HTML basics

Images: 10 points
- images without alt
- alt coverage summary
- do not over-penalize decorative images if they are detectable

Schema: 8 points
- JSON-LD presence
- valid JSON-LD parsing
- invalid JSON-LD should be a fix

Links: 5 points
- internal/external link counts as insight
- do not make links a major score driver

Secondary insights: 7 points
- Open Graph
- Twitter tags
- minor supporting signals

These weights are a guide. If the existing app already has constants, update those constants in one place and keep the model easy to audit.

Score bands:

Good: 80–100
Needs improvement: 50–79
Critical issues: 0–49

Traffic Risk:

High:
- noindex
- missing or empty title
- broken/invalid canonical
- canonical clearly pointing to another URL in a risky way
- missing H1 with other major issues
- serious indexability issue

Medium:
- missing meta description
- weak heading structure
- multiple H1s
- invalid JSON-LD
- many images without alt
- viewport missing
- title/meta length issues

Low:
- Open Graph missing
- Twitter tags missing
- lang missing
- charset missing
- minor link count insights
- social-only metadata
